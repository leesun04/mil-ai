# 🚀 4단계 Git 협업 워크플로우 가이드

본 가이드는 우리 팀의 안전하고 효율적인 코드 관리를 위한 **'Main ➡️ Dev ➡️ 개인 Dev ➡️ Feature'** 4단계 브랜치 전략을 설명합니다. 코드가 꼬이는 것을 방지하기 위해 아래 규칙을 반드시 준수해 주세요!

---

## 🏗️ 1. 브랜치 구조 (Branch Architecture)

우리는 총 4가지 계층의 브랜치를 사용합니다. 위로 올라갈수록 엄격한 규칙이 적용됩니다.

### `main` (최종 배포용)
* 완전히 테스트가 끝나고 에러 없이 돌아가는 최종 코드만 모이는 곳입니다.
* 🚨 **절대 직접 Commit 및 작업 금지!**
* 오직 `dev` 브랜치에서 PR을 통해서만 병합(Merge)합니다.

### `dev` (공용 통합 테스트장)
* `main`으로 가기 전, 각자의 완성된 기능을 모아서 함께 테스트하는 중간 다리입니다.
* 각자의 개인 `dev/[이름]` 브랜치에서 **'하나의 완성된 기능(마일스톤)'** 단위로만 PR을 올립니다.
* 🚨 **PR 생성 후 팀원의 코드 리뷰 및 승인(Approve) 필수!**

### `dev/[본인이름]` (개인 메인 작업실)
* 예: `dev/seonjae`, `dev/jaemin`
* 각자 맡은 파트의 기능을 자유롭게 개발하고 조립하는 개인 공간입니다.
* 작업이 끝난 feature 브랜치들을 이곳으로 모읍니다. (이때는 팀원 확인 없이 스스로 PR & Merge 가능)

### `feature/[기능명]` (1회용 단일 기능 작업대)
* 예: `feature/yolo-bbox`, `feature/llm-prompt`
* 세부적인 단일 기능을 개발할 때 임시로 파는 브랜치입니다.
* 내 개인 방(`dev/[이름]`)으로 PR 및 병합이 완료되면 즉시 삭제합니다.

## 🔄 2. 기본 작업 파이프라인 (Daily Workflow)

### Step 1. 작업 준비 및 기능 브랜치 생성
코딩을 시작할 때는 항상 내 방(`dev/[이름]`)을 최신화한 후 새로운 기능 브랜치를 파생시킵니다.

```bash
# 1. 내 개발 브랜치로 이동
git checkout dev/[내이름]

# 2. 공용 거실(dev)의 최신 코드가 있다면 내 방으로 당겨와서 동기화 (충돌 예방!)
git pull origin dev

# 3. 새로운 기능 작업을 위한 1회용 브랜치 생성 및 이동
git checkout -b feature/[작업할기능명]
```

### Step 2. 코드 작성 및 깃허브 업로드
단일 기능 개발을 완료한 후, 코드를 깃허브 서버에 업로드합니다.

```bash
# 1. 변경된 모든 파일을 장바구니에 담기
git add .

# 2. 커밋 메시지 작성 (아래 커밋 컨벤션 참고!)
git commit -m "feat: [기능 요약 설명]"

# 3. 깃허브 서버로 내 기능 브랜치 업로드
git push origin feature/[작업할기능명]
```

### Step 3. 1차 병합: 내 방으로 가져오기 (Self PR)
자잘한 기능이 완성되었으니 내 메인 작업실로 코드를 합칩니다.

1. 깃허브에서 **[New pull request]** 클릭
2. 🎯 **방향 설정:** `base: dev/[내이름]` ⬅️ `compare: feature/[작업할기능명]`
3. 팀원 리뷰 없이 확인 후 바로 **Merge pull request** 클릭!
4. 완료 후 **Delete branch**를 눌러 다 쓴 feature 브랜치 삭제

### Step 4. 2차 병합: 공용 거실로 결재 올리기 (Team PR)
내 방에 기능들이 모여서 **'하나의 완성된 모듈'**이 되었다면, 재민님과 코드를 섞기 위해 공용 공간에 올립니다.

1. 깃허브에서 **[New pull request]** 클릭
2. 🎯 **방향 설정:** `base: dev` ⬅️ `compare: dev/[내이름]`
3. 🚨 **주의:** 이때는 반드시 재민님에게 리뷰를 요청하고, 코드가 잘 돌아가는지 확인받은 뒤에 Merge를 진행합니다!

## 📝 3. 커밋 메시지 작성 규칙 (Commit Convention)

협업 시 "이 코드가 어떤 코드인지" 직관적으로 파악하기 위해 커밋 메시지 앞에 아래의 **말머리(태그)**를 반드시 붙여주세요.

### 📌 커밋 말머리 종류
* **`feat:`** : 새로운 기능 추가 (예: 모델 연동, 알고리즘 구현 등)
* **`fix:`** : 버그 또는 에러 수정
* **`docs:`** : 문서 수정 (README.md 등)
* **`refactor:`** : 코드 리팩토링 (기능 변화 없이 코드 구조만 개선)
* **`chore:`** : 자잘한 수정, 패키지 매니저 설정 등

### 💡 작성 예시
* `feat: 3D 포인트 클라우드 RANSAC 파라미터 최적화 로직 추가`
* `feat: Streamlit UI 대시보드 기본 레이아웃 구성`
* `fix: YOLO 모델 결과값 좌표 틀어짐 현상 해결`
* `docs: 프로젝트 환경 세팅 가이드 업데이트`
